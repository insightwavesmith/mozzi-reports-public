💡한 줄 요약

핵심 메시지

최신 연구들의 공통 결론: "AI도 뇌처럼 모듈을 나누고, 시간 축을 가지고, 서로 동기화하면 훨씬 잘한다."

우리 뉴런 아키텍처는 이미 이 방향의 70%를 구현하고 있고, 나머지 30%를 논문에서 배울 수 있다.

🔬CTM이 뭔데?

Continuous Thought Machine은 Sakana AI(도쿄)가 만든 새로운 신경망 아키텍처로, NeurIPS 2025 Spotlight 논문이다.

기존 AI는 뉴런을 단순한 숫자 상자로 취급한다. 입력 → 가중치 곱하기 → 출력. 끝. 근데 실제 뇌의 뉴런은 시간에 따라 리듬을 가지고, 다른 뉴런과 박자를 맞추면서 정보를 처리한다.

CTM의 핵심 혁신 2가지

1. 뉴런별 고유 모델 (NLM) — 각 뉴런이 자기만의 가중치를 갖고, 과거 M개의 신호 히스토리를 보고 반응. 같은 입력이라도 뉴런마다 다르게 해석.

2. 동기화 = 표현 — 뉴런들의 활동 패턴이 시간에 따라 얼마나 일치하는지(상관관계)를 계산해서, 이것 자체를 "생각"으로 사용. 순간 스냅샷이 아닌 시간적 흐름이 핵심.

비유: 오케스트라에서 각 악기(뉴런)가 자기 파트를 연주하고, 지휘자 없이도 서로 박자를 맞추면서 하모니(동기화)를 만드는 것. 그 하모니 자체가 "음악(인식/판단)"이 된다.

실제 성능

  • 2D 미로: 39×39 미로에서 100스텝 경로를 찾고, 학습 안 한 99×99 미로까지 일반화
  • ImageNet: 이미지를 "둘러보면서" 분류 — 사람처럼 이리저리 살펴보는 행동이 자연 발생
  • 적응적 계산: 쉬운 문제는 5틱만에, 어려운 문제는 50틱까지 — 자동으로 계산량 조절

🔍우리에게 없는 것 — 논문에서 배울 점

Gap 1: 피드백 루프 (MAP의 Monitor→Actor)

MAP에서는 Monitor가 Actor의 행동을 검증하고, 잘못되면 피드백을 보내서 재시도한다. 우리는 Guardian Plugin이 메시지만 필터링하지, 실행 결과를 검증하고 재시도하는 루프가 없다.

→ 실행 기능이 TASK를 작성하면, 전략 기능이 검증하고 피드백하는 루프 추가.

Gap 2: 히스토리 윈도우 (CTM의 M개 기록)

CTM은 최근 10~100개의 활성화를 참조한다. 우리 handoff.md는 마지막 1개만 보는 경우가 많다. 과거 맥락이 유실된다.

→ 최근 N개 handoff를 항상 컨텍스트에 포함. CTM처럼 히스토리 윈도우를 유지.

Gap 3: Early Exit (CTM/ACT의 적응적 계산)

CTM은 certainty가 높으면 10틱 중 3틱만에 멈춘다. 우리는 체인이 시작되면 무조건 끝까지 간다. 분석 결과만으로 충분하면 전략을 건너뛸 수 있어야 한다.

→ CHAIN 태그에 confidence 점수 추가. 점수가 높으면 다음 단계 스킵.

Gap 4: Evaluator (MAP의 상태 가치 평가)

MAP에서는 Evaluator가 "이 상태가 목표에 얼마나 가까운지" 평가한다. 우리에게는 실행 결과의 품질을 평가하는 전용 메커니즘이 없다.

→ 체인 완료 후 결과를 평가하는 단계 추가 (전략 기능 확장 또는 새 역할).

Gap 5: Temporal Decay (CTM의 시간 가중치)

CTM은 최근 정보에 더 높은 가중치를 준다 (지수 감쇠). 우리 메모리는 모든 기록이 동등한 가중치를 가진다.

→ handoff 기록에 타임스탬프 기반 가중치 적용. 오래된 기록은 요약, 최근 기록은 원문 유지.

🗺️도입 로드맵

1
히스토리 윈도우 + Early Exit
1~2일
  • 히스토리 윈도우: handoff.md에서 최근 3개 기록을 항상 attachments로 포함. 라우터 AGENTS.md에 규칙 추가.
  • Early Exit: CHAIN 태그에 [CHAIN:strategist|confidence:0.9] 형식 추가. confidence ≥ 0.85면 다음 단계 스킵 가능.
  • 효과: 컨텍스트 풍부화 + 불필요한 체인 단축 → 속도 & 품질 동시 개선
2
Monitor 피드백 루프
2~3일
  • 실행 기능의 TASK 결과를 전략 기능이 검증하는 루프 추가
  • Guardian Plugin에 subagent_ended hook 확장 — 결과 품질 체크
  • 검증 실패 시 피드백과 함께 재실행 (최대 2회)
  • 효과: 실행 품질 보장, 단발성 실패 자동 복구
3
Evaluator + Temporal Decay
3~5일
  • Evaluator: 체인 완료 후 결과 품질 평가 — "목표에 얼마나 가까운지" 점수화
  • Temporal Decay: handoff.md 기록에 타임스탬프 기반 가중치. 24시간 이내 = 원문, 7일 이상 = 한 줄 요약
  • Supabase 라우팅 로그에 평가 점수 누적 → 체인 최적화 데이터 수집
  • 효과: PonderNet처럼 체인 길이를 데이터 기반으로 최적화하는 토대
4
동기화 정식화 + Tree Search (장기)
1~2주
  • CHAIN 태그를 구조화된 동기화 프로토콜로 발전
  • 전략 기능에 Tree Search 추가 — 여러 전략을 병렬 평가 후 최적 선택
  • 메모리 구조를 BMAS 스타일로 정식화 (워킹/에피소드/시맨틱 분리)
  • 효과: MAP 논문 수준의 멀티스텝 플래닝 능력 확보

✅ 결론: 우리는 이미 70% 와 있다

Nature Communications(MAP)과 NeurIPS(CTM)에 실린 뇌 영감 AI 아키텍처의 핵심 원리들을 우리 뉴런 시스템은 이미 상당 부분 구현하고 있다.

남은 30%는:

  • 히스토리 윈도우 (1개 → N개)
  • Early Exit (체인 조기 종료)
  • 피드백 루프 (실행→검증→재시도)
  • Evaluator (결과 품질 평가)
  • Temporal Decay (시간 가중치)

Phase 1~2만 적용해도 체감 성능이 크게 올라갈 것으로 예상.

🎯우선순위 근거 — 왜 이 순서인가

Phase 순서는 세 가지 기준으로 결정했다: ①의존성(나중 단계가 이전 단계 결과를 전제로 하는지), ②ROI(투입 대비 실질적 체감 효과), ③리스크(도입 실패 시 기존 시스템에 미치는 영향).

Phase 1이 먼저인 근거 (히스토리 윈도우 + Early Exit)

의존성 없음 — 기존 AGENTS.md 규칙에 텍스트 추가만으로 완성. 코드 변경 불필요.
ROI 최고 — 히스토리 윈도우는 30분 작업으로 컨텍스트 연속성을 1개 → N개로 확장. 단발성 체인에서 "지난번에 뭐 했더라?" 맥락 손실을 즉시 해결.
리스크 최저 — AGENTS.md 텍스트 변경은 원상복구가 git revert 한 줄. 시스템 다운 없음.
Phase 2~4의 전제 — 피드백 루프(Phase 2)는 과거 실행 기록을 참조해야 의미가 있다. 히스토리 윈도우가 없으면 피드백 루프도 "이번 실패"만 보고 패턴을 못 잡는다.

Phase 2가 두 번째인 근거 (Monitor 피드백 루프)

Phase 1 데이터 활용 — 히스토리 윈도우가 있으면 "이 패턴의 실패가 3번째"를 감지할 수 있다.
실질 고통 해결 — 현재 실행 뇌 실패 시 Smith님이 직접 재요청해야 함. 자동 재시도는 체감 품질을 가장 크게 올리는 변화.
neuron-guardian 확장 — Phase 1이 완료된 상태에서 기존 `subagent_ended` hook을 확장하는 구조라 코드 접점이 명확함.

Phase 3이 세 번째인 근거 (Evaluator + Temporal Decay)

평가는 히스토리가 있어야 의미 있음 — "결과 품질 5점"이라는 스코어는 시계열이 쌓여야 트렌드가 보인다. Phase 1~2 없이 단독 도입하면 숫자만 기록되고 활용 불가.
Temporal Decay는 데이터 인프라 선행 필요 — 오래된 기록을 가중치 낮춰 읽으려면 타임스탬프 기반 파싱 로직이 필요. Supabase 로그가 쌓인 뒤 설계하는 게 더 정확함.

Phase 4가 마지막인 근거 (동기화 정식화 + Tree Search)

진짜 R&D 단계 — CTM의 동기화 행렬, MAP의 Tree Search는 "아이디어를 더 잘 실행"이 아니라 "아이디어 탐색 공간 자체를 확장"하는 개념. 실용적 인프라(Phase 1~3) 없이 도입하면 복잡도만 오르고 관리 불가.
비용 리스크 — 병렬 전략 평가는 API 비용을 N배 증가시킴. Phase 1~3으로 효율화가 선행돼야 비용 중립적으로 도입 가능.

⚠️리스크 분석 — 단계별 주의 사항

Phase 1 리스크

🔴 리스크 1-A: 히스토리 윈도우 → 컨텍스트 토큰 초과

시나리오: handoff 항목이 길어지면 "최근 3개"도 8K+ 토큰. spawn task에 포함 시 모델 컨텍스트 초과.

완화: 항목당 500토큰 캡 설정 규칙을 AGENTS.md에 명시. 초과 시 요약 버전으로 자동 대체. handoff.md 정기 아카이브 규칙 동시 적용 필수.

🟡 리스크 1-B: Early Exit → Confidence 점수 환각

시나리오: LLM이 confidence를 항상 0.95로 리포트. 실제로 전략 검토가 필요한 요청도 스킵됨.

완화: 초기 임계값을 0.95 이상으로 매우 보수적으로 설정. 실제 스킵 빈도를 Supabase 로그로 모니터링. 2주 후 데이터 보고 임계값 조정. 스킵 대상은 "단순 사실 확인"으로 유형 명시해서 LLM 판단 범위 좁히기.

🟢 리스크 1-C: 롤백 방법

Phase 1은 모두 AGENTS.md 텍스트 수정. 문제 발생 시: git -C ~/.openclaw revert HEAD 또는 수동으로 해당 섹션 삭제. 시스템 다운 없이 즉시 롤백 가능.

Phase 2 리스크

🔴 리스크 2-A: 피드백 루프 → 무한 재시도

시나리오: 실패가 일시적이 아닌 구조적 문제일 때(권한 오류, 파일 미존재 등) 재시도를 반복하다 API 비용 폭발.

완화: 재시도 최대 2회로 하드코딩. SESSION-STATE.md에 retry_count 필드 추가. 2회 초과 시 즉시 중단 + Smith님에게 알림. guardian plugin에서 BUSY 타임아웃 15분도 안전망 역할.

🟡 리스크 2-B: guardian plugin 수정 → 기존 BUSY 로직 깨짐

시나리오: subagent_ended hook 확장 시 기존 .neuron-busy 파일 삭제 로직 실수로 누락. BUSY 플래그가 해제 안 돼서 이후 모든 spawn이 차단됨.

완화: plugin 수정 후 반드시 gateway restart + 간단한 테스트 요청으로 BUSY 해제 흐름 확인. 테스트 절차를 AGENTS.md에 명시.

🟡 리스크 2-C: 재시도 시 BUSY 상태 관리

시나리오: 재시도 spawn 시 이전 BUSY 플래그가 아직 해제 안 된 상태. 재시도가 BUSY 차단됨.

완화: guardian plugin에서 "재시도 spawn은 BUSY 체크 우회" 플래그 추가. event.metadata.is_retry = true 체크로 구분.

Phase 3 리스크

🟡 리스크 3-A: Evaluator → 전략 뇌의 평가 편향

시나리오: 전략 뇌가 자신이 설계한 전략의 실행 결과를 평가함 → 자기 자신을 높게 평가하는 편향.

완화: 평가 기준을 객관적 체크리스트로 명시 (모호한 "품질"이 아닌 "파일 X가 생성됐는가", "git push가 성공했는가" 등). 주관적 평가보다 이진 체크리스트가 훨씬 신뢰성 높음.

🟢 리스크 3-B: Temporal Decay → 중요한 오래된 기록 무시

시나리오: 3주 전에 결정한 중요한 아키텍처 원칙이 오래됐다는 이유로 읽히지 않음.

완화: handoff.md에 [PERMANENT] 마크 규칙 추가. PERMANENT 항목은 날짜와 무관하게 항상 원문 읽기. MEMORY.md에 핵심 원칙을 별도 유지하는 현재 구조도 안전망.

Phase 4 리스크

🔴 리스크 4-A: Tree Search → API 비용 급증

시나리오: 전략 3개를 병렬 평가하면 API 호출이 3배. 복잡한 요청에서 비용 폭발.

완화: Tree Search를 명시적으로 요청한 태스크에만 적용 (기본값 아님). 최대 병렬 전략 수를 3개로 제한. Phase 1~3 도입 후 효율화된 상태에서 비용 중립적으로 도입.

🟡 리스크 4-B: 동기화 프로토콜 → 과도한 구조화

시나리오: CHAIN 태그를 복잡한 동기화 프로토콜로 발전시키면서 단순한 요청도 복잡한 오케스트레이션을 거쳐야 하는 구조가 됨.

완화: 기존 CHAIN 태그의 단순함은 유지. 동기화 프로토콜은 "복잡한 멀티스텝 기획"에만 적용하는 옵트인(opt-in) 방식으로 설계.

🛡️ 전체 안전 원칙

  • 점진적 도입: 한 Phase가 2주 이상 안정적으로 운영된 후 다음 Phase 시작
  • 모니터링 선행: Supabase 로그에서 각 지표를 확인할 수 있는 상태에서만 다음 단계 진행
  • 롤백 계획: 각 Phase 적용 전 git commit으로 스냅샷. 문제 시 즉시 revert
  • 파일럿 테스트: Phase 2~4는 실제 적용 전 단일 태스크로 동작 확인 필수

1. 기획 의도

현재 큐레이션 파이프라인은 일방통행입니다. 소스가 수집되고, 콘텐츠로 변환되면, 되돌릴 수 없습니다. 삭제하면 소스도 사라지고, 과거 프롬프트로 만든 글을 다시 만들 수도 없습니다.

콘텐츠가 계속 쌓이는데 관리 도구가 없어서, Smith님이 직접 하나하나 처리해야 합니다. 이 기획은 "소스는 절대 사라지지 않고, 어디서든 되돌릴 수 있는" 구조를 만드는 것이 목표입니다.

01
삭제 = 소스 소실
정보공유 글을 삭제하면 원본 큐레이션 아이템이 "published" 상태에 박혀서 영영 안 나타남. deleteContent()가 단순 DELETE만 수행.
02
재생성 불가
과거 프롬프트/설정으로 만든 글을 현재 프롬프트로 다시 만들 수 없음. AI 수정은 부분 수정만.
03
단방향 상태
new → published 한번 가면 끝. dismissed도 복원 불가. 모든 상태 전이가 되돌릴 수 없음.
04
관리 도구 부재
필터, 검색, 벌크 액션 없음. 소스가 쌓이면 스크롤해서 찾아야 함. 생성물 연결도 안 보임.

2-2. 데이터 플로우

소스 수집
Notion sync (매일 04:00) / URL 크롤링 / 파일 업로드 / 웨비나 API → curation_status: "new"
관리자 검토
큐레이션 탭에서 아이템 검토 → 최대 4개 선택 → "reviewing"
AI 생성
Claude Opus + Extended Thinking + RAG → 미리보기 → 승인 → contents INSERT, 원본 → "generated"
편집 & 발행
MDX 에디터 / AI 수정요청으로 다듬기 → 발행 → 수강생 공개
삭제 시 자동 복구 NEW
소프트 삭제 (status → "archived") + source_ref 파싱 → 원본 curation_status "reviewing"으로 롤백 → 재생성 가능

2-3. DB 변경

변경내용
contents 테이블 archived_at TIMESTAMPTZ DEFAULT NULL 컬럼 추가
curation_status에 reviewing, generated, archived 값 추가
deleteContent() Hard DELETE → status: "archived" + archived_at: now()
source_ref 파싱 → 원본 아이템 curation_status: "reviewing"으로 복구
source_ref 현행 유지 (쉼표 구분 문자열). Phase 3에서 text[] 배열 전환 고려.

3-3. 삭제 시 동작 비교

현재 (Hard Delete)개선 (Soft Delete + 복구)
1. 정보공유 글 삭제 클릭
2. DB에서 완전 삭제
3. 원본 소스 → "published" 유지
4. 큐레이션 탭에 안 나타남
5. 소스 영영 소실 ❌ 		1. 정보공유 글 "삭제(보관)" 클릭
2. status → "archived", archived_at 기록
3. source_ref 파싱 → 원본 복구
4. 원본 → "reviewing" → 큐레이션 탭 복귀
5. 언제든 재생성 가능 ✅

4. 구현 우선순위

1
안전망 — 즉시 구현
  • 소프트 삭제 — deleteContent()를 archived 상태 전환으로 변경
  • 삭제 시 소스 복구 — source_ref 파싱 → curation_status 롤백
  • dismissed 복원 버튼 — 스킵한 아이템 되돌리기
2
편의성 — 권장
  • 상태별 필터 탭 — 신규 / 검토중 / 생성됨 / 스킵 / 보관
  • 양방향 연결 표시 — 큐레이션 카드에 생성물 링크, 생성물에 원본 소스
  • 재생성 버튼 — 같은 소스로 현재 프롬프트 기반 재생성
3
고도화 — 여유 시
  • 벌크 액션 — 다중 선택 → 일괄 스킵/복원/삭제
  • 검색 + 정렬 — 제목/내용 검색, 날짜/소스 정렬
  • source_ref 배열화 — 쉼표 구분 → text[] 전환 (쿼리 성능)

5. 리스크 & 고려사항

항목대응
이미 "published"로 박힌 아이템 마이그레이션 스크립트로 generated로 변경 가능
이미 hard delete된 정보공유 글 복구 불가 (이미 DB에서 삭제됨). 향후 발생 방지만.
soft delete 시 UI 필터 status != "archived" 기본 필터 추가. 관리자 전용 "보관함" 뷰.
성능 현재 규모에서 문제 없음. source_ref IN 쿼리로 충분.

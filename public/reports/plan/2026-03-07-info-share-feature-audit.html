📐 정보공유 전체 구조

정보공유는 크게 큐레이션 파이프라인(소스 수집 → AI 생성)과 정보공유 탭(발행된 글 관리)으로 나뉨

소스 수집 → 사이드바(소스별 필터) → 큐레이션 뷰(인박스/토픽맵) 또는 커리큘럼 뷰(블루프린트/자사몰사관학교)

큐레이션 뷰에서 콘텐츠 선택 → 정보공유 생성(AI Opus) → 미리보기/수정 → 초안 저장 → 콘텐츠 탭에서 게시

게시된 정보공유 → 정보공유 탭(테이블 뷰) + 수강생 페이지(/posts)

🚨 오류 및 미구현 사항

P1 #1. 커리큘럼 잠금 — 유연성 부족

현상: 커리큘럼 뷰에서 순차 발행이 강제됨. 중간 강의를 건너뛰고 뒤쪽 강의를 먼저 발행할 수 없음.
원인getPublishStatuses()가 위에서부터 순서대로 스캔하면서 첫 번째 미발행 항목만 "다음 발행"으로 마킹. 나머지는 무조건 "잠금".
영향관리자가 특정 강의를 우선 발행하고 싶을 때 불가능. 순서대로만 작업해야 함.
파일src/components/curation/curriculum-view.tsx L52~66
제안① "잠금 해제" 버튼 추가 (관리자가 수동으로 특정 항목 발행 가능하게)
② 또는 잠금 로직 자체를 제거하고 모든 항목에 "정보공유 생성" 버튼 노출

P1 #2. 스킵 → 신규 되돌리기 미구현

현상: 스킵(dismissed) 처리한 콘텐츠를 다시 신규(new)로 되돌리는 기능이 없음. 실수로 스킵하면 복구 불가.
원인batchUpdateCurationStatus()의 타입이 "selected" | "dismissed" | "published"만 허용. "new"가 유니온에 없음.
스킵 탭 UI에도 "되돌리기" 버튼 없음.
영향운영 중 실수로 스킵한 콘텐츠를 복구하려면 DB 직접 수정 필요.
파일src/actions/curation.ts L214, L237
src/components/curation/curation-view.tsx (스킵 탭 렌더링 부분)
제안① batchUpdateCurationStatus 타입에 "new" 추가
② 스킵 탭 카드에 "신규로 되돌리기" 버튼 추가

P2 #3. 소스 전환 시 매번 API 호출 (캐싱 없음)

현상: 사이드바에서 소스를 전환할 때마다 loadContents() → Supabase API 재호출. 같은 소스를 다시 선택해도 캐시 없이 전체 리로딩.
원인useEffect 의존성에 sourceFilter가 포함되어 있어, 전환 때마다 무조건 새 API 호출.
클라이언트 캐시 레이어 없음.
영향UX 느려짐. 소스 간 왔다갔다 하면서 비교할 때 매번 로딩 스피너.
파일src/components/curation/curation-view.tsx L60~92 (loadContents)
제안① React Query / SWR 도입으로 클라이언트 캐싱
② 최소한 useRef 기반 간이 캐시 (sourceFilter → 데이터 맵)

P1 #4. "발행됨" 탭에 미처리/처리 혼재

현상: "발행됨" 탭에 정보공유가 실제로 게시(published) 완료된 것과, AI 생성 → 초안만 만들어진 것이 구분 없이 같이 보임.
원인createInfoShareDraft()의 4단계에서 원본 소스 콘텐츠의 curation_status를 "published"로 변경함.
이 의미는 "이 소스로 정보공유를 만들었다"인데, 실제로는 info_share 자체가 아직 draft 상태일 수 있음.

"발행됨" 탭 카운트 = curation_status=published인 원본 소스 수. 하지만 이 중 일부는 info_share가 draft인 채로 남아있을 수 있음.
영향"발행됨 N건"이라고 표시되지만 실제 수강생이 볼 수 있는 게시 완료된 정보공유 수와 다를 수 있음. 관리자 혼란.
파일src/actions/curation.ts L320~335 (createInfoShareDraft)
src/components/curation/curation-view.tsx (발행됨 탭 필터)
제안① 원본 상태를 "used" 또는 "generated"로 분리 (info_share 실제 게시 시에만 "published")
② 또는 "발행됨" 탭에서 linked_info_shares의 실제 status(draft/published)를 같이 표시

📊 curation_status 상태 흐름

new → (관리자 선택) → selected → (정보공유 생성) → published
new → (관리자 스킵) → dismissed → ❌ new로 되돌리기 불가

문제: published의 의미가 "info_share가 게시됨"인지 "info_share 초안이 생성됨"인지 모호.
실제로는 createInfoShareDraft() 호출 시점에 바로 published로 변경됨 → 초안 생성 = published 처리.

🔍 경쟁사 분석기 UX 리디자인

개발 + 고객 관점 통합 기획서 — 2026.03.08

🎯 왜 리디자인하는가

수강생이 이 기능을 쓰는 이유: "효과 좋은 경쟁사 광고 소재를 빠르게 찾아서 내 것으로 만들기"

지금은 "검색은 되지만 쓰기 불편한" 상태:

  • 30건 고정 — 필터 걸면 건수가 더 줄어듦
  • 다운로드가 개별 1건씩 — 레퍼런스 수집에 시간 오래 걸림
  • 경쟁사 고정 약함 — 새 광고 올라와도 알 수 없음
  • 소재 유형 분류 부족 — 캐러셀(슬라이드) 구분 없음
  • 검색이 키워드 의존 — 정확한 브랜드 매칭 어려움
  • "Meta에서 보기" 링크 동작 안함 — 인스타/페북 연결 필요

👤 수강생 사용 여정

① 경쟁사 찾기 (검색) └→ "올리브영" 검색 → 브랜드 목록 (프로필+인스타+좋아요수) └→ 또는 인스타 URL / 랜딩페이지 URL 붙여넣기 └→ 또는 한글 브랜드명 → 영어 인스타 계정도 인식 ② 경쟁사 핀 고정 (모니터링) └→ 검색 결과에서 브랜드 선택 → 📌 핀 등록 └→ 매일 접속 → 🔴 NEW +3건 표시 → 클릭하면 새 광고만 보기 ③ 광고 분석 (필터 + 정렬) └→ 30일+ 필터 → "오래 돌아간 광고 = 효과 좋은 광고" └→ 이미지 / 슬라이드 / 영상 분류 └→ 운영기간순 정렬 → 상위 = 가장 효과 좋은 소재 └→ 더보기 → 전체 광고 무제한 탐색 ④ 소재 다운로드 (레퍼런스 수집) └→ ☑️ 원하는 카드 체크 → "선택 다운로드 (ZIP)" └→ 이미지+영상 모두 포함 (클라이언트 직접 다운로드) └→ 슬라이드 → 전체 이미지 폴더로 묶음 ⑤ 광고 원본 확인 └→ 카드에서 인스타/페북 링크 → 원본 광고 페이지 이동

📥 축 4: 다운로드 UX 신규

핵심 변경: 클라이언트 직접 다운로드 (서버 안 탐)

기존 문제: Vercel 서버가 fbcdn에서 파일 → 클라이언트 전달 → 영상 5~30MB → 서버 응답 제한(~5MB) 초과.
해결: 브라우저가 fbcdn에서 직접 다운로드 → Vercel 서버 비용 0, 용량 제한 없음.
✅ CORS 검증 완료: fbcdn 응답 헤더에 Access-Control-Allow-Origin: * 확인. 이미지+영상 모두 브라우저에서 직접 fetch 가능.
결론: 서버 프록시 불필요. /api/competitor/download 삭제 가능 (또는 극단적 fallback으로만 유지).

// 개별 다운로드 (CORS 가능 시) const res = await fetch(fbcdnUrl); const blob = await res.blob(); const url = URL.createObjectURL(blob); const a = document.createElement('a'); a.href = url; a.download = filename; a.click(); // CORS 불가 시 → 서버 프록시 fallback const res = await fetch(`/api/competitor/download?url=${encodeURIComponent(fbcdnUrl)}`); // ZIP (클라이언트) const zip = new JSZip(); for (const ad of selectedAds) { const blob = await fetch(ad.imageUrl).then(r => r.blob()); zip.file(filename, blob); } const content = await zip.generateAsync({ type: 'blob' }); saveAs(content, 'competitor-ads.zip');

    Vercel 서버 프록시 대신 브라우저에서 직접 fbcdn 다운로드. 영상 용량 제한 해소 + 서버 비용 0. CORS 안 되면 서버 fallback 유지.

    키워드 검색 → 바로 광고 목록이 아니라, 먼저 브랜드를 정확히 선택한 뒤 해당 page_id의 광고만 조회. 검색 정확도 대폭 향상.
  3. 모니터링 = page_id 기반 + NEW 자동 감지
    수강생이 매일 확인하는 루틴. 새 광고 올라오면 즉시 알 수 있어야 함. Cron 1일 1회로 크레딧 절약.
열린 질문: 수강생별로 모니터링 브랜드가 다른데, Cron에서 전체 사용자의 브랜드를 합산하면 크레딧 소비가 커질 수 있다. 중복 page_id는 1회만 호출하는 최적화가 필요할까, 아니면 사용자별 개별 호출이 맞을까?

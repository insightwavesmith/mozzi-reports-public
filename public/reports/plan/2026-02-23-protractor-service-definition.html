A

이 서비스가 뭔지 (비개발자용)

한 줄 정의

"수강생의 Meta 광고 성과를 동기 전체와 비교하여 어디서 새고 있는지 자동으로 진단해주는 광고 진단 도구"

누가 쓰는지

  • 수강생: 자신의 광고계정 ID를 입력하면 오늘 돌아간 광고가 동기 대비 어떤 수준인지 즉시 확인
  • 관리자 (Smith님): admin.html에서 수강생 계정 등록·삭제·수정, 데이터 동기화 트리거, 연동 상태 점검

왜 필요한지

수강생이 Meta 광고관리자를 봐도 "내 광고가 잘 되고 있는 건지 아닌 건지" 판단 기준이 없다.

이 서비스는 동기 전체(같은 Meta 광고 토큰으로 접근 가능한 모든 계정)를 기준으로 벤치마크를 만들고, 수강생 개인의 광고를 그 기준과 비교해서 🟢🟡🔴 3단계 판정을 내린다.

B

데이터 흐름 (처음부터 끝까지)

0단계: 계정 등록 (Admin → BigQuery)

관리자가 admin.html에서 수강생 정보 등록:

  • Meta 광고계정 ID (act_xxx 형식)
  • Mixpanel 프로젝트 ID + 시크릿
  • Mixpanel 보드 ID (대시보드 링크 연결용)

이 정보는 BigQuery accounts 테이블에 저장된다. 등록된 계정만 일일 데이터 수집 대상이 된다.

1단계: 매일 자동 수집 (collect_daily.py)

매일 Cloud Scheduler가 /collect-daily 엔드포인트를 호출한다.

1-1. Meta 광고 데이터 수집

  • BigQuery accounts 테이블에서 active = TRUE 계정 목록 조회
  • 각 계정에 Meta Graph API 호출 → "어제" 날짜의 활성 광고 전부 가져옴
  • 광고별로 27개 지표 계산:
    • 영상 지표: 3초 시청률 / ThruPlay율 / 지속비율
    • 참여 지표: 좋아요·댓글·공유·저장 (만 노출당)
    • 전환 지표: CTR / 클릭→장바구니 / 클릭→결제시작 / 결제→구매 / 클릭→구매 / 도달→구매 / ROAS
    • 랭킹: Meta가 부여하는 quality/engagement/conversion ranking (ABOVE_AVERAGE, AVERAGE, BELOW_AVERAGE)
  • 결과를 BigQuery daily_ad_insights 테이블에 저장

1-2. Mixpanel LP 데이터 수집

  • 같은 accounts 테이블에서 Mixpanel 인증정보가 있는 계정만 대상
  • Mixpanel Segmentation API를 호출, utm_source 필터로 Meta 광고 트래픽만 추출
  • 수집 지표:
    • 방문자 수 (view_product 이벤트, Meta UTM 소스 필터)
    • 이탈률 계산법 (V7): page_exit 이벤트의 duration 분포 조회 → 1초/10초 미만 이탈률 (음수 방지)
    • 스크롤 깊이: 25% / 50% / 75% / 100% 도달률
    • 전환율: LP세션→장바구니 / LP세션→결제시작 / LP세션→구매 / 결제→구매
    • 10초 체류율, Meta 채널 결제시작 기반 구매 추정치
  • 결과를 BigQuery daily_lp_metrics 테이블에 저장
2단계: 매주 벤치마크 갱신 (collect_benchmarks.py)

매주 월요일 오전 9시 Cloud Scheduler가 자동 실행. "전체 수강생 광고를 모아서 평균 기준선을 만드는 과정" — Smith님이 가장 깊게 설계한 핵심 로직.

2-1. 전체 계정 광고 수집 (벤치마크 모집단)

  • Meta API로 접근 가능한 모든 광고계정 조회 (Smith님 토큰으로 접근 가능한 전체)
  • EXCLUDED_ACCOUNTS 5개 계정 제외 (특정 계정은 벤치마크 집계에서 제외)
  • 각 계정별로 최근 7일 데이터, 노출 3,500회 이상인 광고만 포함
  • 계정당 지출 상위 10개 광고만 선별 (품질이 높은 활성 광고 위주)
  • 결과를 ad_insights_classified 테이블에 저장 (벤치마크 원본 데이터)

2-2. 벤치마크 통계 계산 (27개 그룹)

27개 그룹 = 크리에이티브 타입 3개 × 랭킹 종류 3개 × 등급 3개

크리에이티브랭킹 종류등급
VIDEOquality (품질)ABOVE_AVERAGE
IMAGEengagement (참여)AVERAGE
CATALOGconversion (전환)BELOW_AVERAGE

각 그룹별로 단순평균(V8부터 변경, 기존 가중평균) 계산 → avg_video_p3s_rate, avg_thruplay_rate, avg_ctr, avg_click_to_cart_rate 등 15개 지표

2-3. MEDIAN_ALL 벤치마크 계산 (6개 그룹)

판정 기준의 "중간값" 역할을 하는 전체 단순평균. 27개 그룹과 별도로, 크리에이티브 타입 3개 × engagement/conversion 2개 = 6개 그룹만 계산. ranking_group = 'MEDIAN_ALL'로 저장.

판정 로직에서 사용하는 기준선:

  • ABOVE_AVERAGE 값 = 우수 기준선 (이 이상이면 🟢)
  • ABOVE_AVERAGE × 75% = 최저 기준선 (이 미만이면 🔴)

2-4. LP 벤치마크 계산 (우수 계정만 포함)

  • LP 벤치마크 자격 조건: 최근 7일 광고 중 70% 이상이 ABOVE_AVERAGE 품질이고 AVERAGE 이상 광고 2개 이상인 계정만
  • 자격 통과 계정의 Mixpanel에서 7일치 LP 데이터 수집
  • 해당 계정들의 단순평균 → lp_benchmarks 테이블 저장
3단계: 수강생 대시보드 조회 (dynamic.html ↔ dashboard_api.py)

수강생이 dynamic.html에 광고계정 ID를 입력하면:

  1. API 호출: GET /?account_id=1234567&[date or start_date+end_date]
  2. 서버가 처리:
    • BigQuery daily_ad_insights에서 수강생 광고 데이터 조회
    • BigQuery daily_lp_metrics에서 LP 데이터 조회 (있으면)
    • BigQuery benchmarks 테이블에서 최신 벤치마크 조회
    • 서버에서 🟢🟡🔴 판정 계산 (수식 노출 방지를 위해 클라이언트에서 처리 안 함)
  3. 화면 표시: 판정 결과 + 기준선 값이 이미 포함된 JSON 응답
∠

총가치 3요소와 코드의 연결점

총가치 = 입찰가 × 추산행동률 + 광고품질  (Meta 경매 공식)

광고품질 (Ad Quality)

  • Meta가 직접 부여하는 quality_ranking 필드
  • ABOVE_AVERAGE / AVERAGE / BELOW_AVERAGE 3등급
  • 코드에서: normalize_ranking(insight.get("quality_ranking"))
  • 대시보드에서: 광고 카드의 "품질" 박스 🟢🟡🔴

추산행동률 (Estimated Action Rate)

Meta가 두 가지로 측정:

  • 참여 추산행동률: engagement_ranking → 얼마나 많은 사람이 좋아요·댓글·공유 등 반응할 것으로 예측되는가
  • 전환 추산행동률: conversion_ranking → 얼마나 많은 사람이 구매 전환할 것으로 예측되는가

이 두 랭킹이 벤치마크 27개 그룹의 핵심 분류 기준이다. ABOVE_AVERAGE 광고들의 평균 지표가 "우수 기준선"이 된다.

입찰가 (Bid Price)

코드에서 직접 입찰가를 다루진 않지만, CTR(클릭률)이 낮으면 Meta 경매에서 밀려나는 구조이므로 CTR / CPM / 지출 지표를 통해 간접 측정된다.

C

핵심 기능 목록 (코드 확인 기반)

✅ 벤치마크 비교 활성

  • 파일: collect_benchmarks.py + dashboard_api.py
  • 방식: 전체 수강생 광고를 VIDEO/IMAGE/CATALOG × engagement/conversion × ABOVE/AVERAGE/BELOW 27개 그룹으로 분류, 단순평균으로 기준선 계산
  • 조회: 대시보드 API 호출 시 benchmarks 테이블에서 최신 날짜 데이터 로드
  • Fallback: CATALOG 샘플이 5개 미만이면 IMAGE 벤치마크로 자동 대체, IMAGE도 부족하면 VIDEO 사용

✅ 🟢🟡🔴 판정 로직 활성

  • 파일: dashboard_api.py → get_verdict() 함수
  • 기준:
    • 🟢 우수: value >= above_avg (ABOVE_AVERAGE 그룹 평균 이상)
    • 🟡 보통: above_avg × 0.75 <= value < above_avg
    • 🔴 미달: value < above_avg × 0.75
    • 낮을수록 좋은 지표(이탈률): 반대 방향 적용
  • 보안: 수식은 서버에서만 계산, 클라이언트에는 결과(v/e 키)만 전송

✅ TOP 5 광고 카드 활성

  • 파일: dashboard_api.py → get_top5_ads(), get_top5_ads_range()
  • 기준: 지출(spend) 기준 상위 5개
  • 표시 내용:
    • Meta 광고관리자 바로가기 링크
    • Mixpanel LP 분석 바로가기 링크 (광고 ID로 필터링된 보드)
    • 품질/참여/전환 3대 판정 요약 박스
    • 영상 지표 테이블 (3초 시청률, ThruPlay율, 지속비율, CTR)
    • 참여 지표 테이블 (좋아요·댓글·공유·참여 합계 — 실제 건수 / 기대 건수 비교)
    • 전환 지표 테이블 (클릭→장바구니, 클릭→결제, 결제→구매, 클릭→구매, 도달→구매, ROAS)
    • ROAS 멀티플 분석

✅ ROAS 멀티플 분석 활성

  • 파일: dynamic.html → renderRoasMultipleAnalysis() 함수
  • 방식: 벤치마크 미달 지표를 벤치마크 수준으로 올렸을 때 ROAS가 몇 배가 되는지 누적 계산
    • CTR 미달 → CTR 개선 시 ROAS 몇 배
    • 클릭→결제시작 미달 → 개선 시 ROAS 추가 몇 배
    • 결제→구매 미달 → 개선 시 ROAS 추가 몇 배
    • 모두 벤치마크 달성 시 최종 ROAS 예측값 표시
  • 단계별 멀티플 공식: 벤치마크값 / 현재값
  • 기간 평균 뷰에서도 동일하게 렌더링됨

⚠️ 퍼널 파이프라인 UI 데이터 미연결

  • 파일: dynamic.html → renderFunnelPipeline() 함수, 관련 CSS 클래스 완비
  • UI: CTR → 10초 체류 → 결제시작 → 구매를 "수도관"처럼 시각화, 각 단계별 이탈 비율 표시
  • 현재 상태: 프론트엔드 렌더링 코드는 완성되어 있으나, 백엔드(dashboard_api.py)가 funnel 키를 응답에 포함하지 않아 현재 렌더링 안 됨
    • renderAverageDashboard() 안에 if (data.funnel) { html += renderFunnelPipeline(data.funnel); }가 있으나 API 응답에 funnel이 없음
    • UI 코드는 있고 데이터 파이프라인은 부분 있지만 API 연결이 끊어진 상태

⛔ LP 품질/전환율 섹션 비활성화

  • 수집 상태: collect_daily.py에서 매일 Mixpanel 데이터 수집 → daily_lp_metrics 테이블 정상 저장 중
  • API 상태: get_lp_quality_data() 함수 구현 완료, API 응답에 lp_quality 키로 포함

// LP 품질/전환율 섹션 숨김 (데이터 정합성 이슈로 비활성화)
// if (data.lp_quality) { ... }
  • 비활성 이유: 데이터 정합성 이슈 (이탈률 음수 발생, V7에서 수정했으나 아직 미복원)

✅ 피드백반 순위 별도 페이지

  • 파일: dashboard_api.py → get_feedback_ranking_data(), feedback-ranking.html
  • 대상: FEEDBACK_MEMBERS에 하드코딩된 6명 (서현석, 성용협, 이민규, 허유범, 정민규, 현명석)
  • 순위 공식: 자사몰 매출 성장률 50% + 광고매출 성장률 50% (정규화 후 100점 만점)
  • 데이터 소스: 자사몰 매출 = Mixpanel purchase 이벤트 payment_amount 합계, 광고매출 = Meta API 계정 레벨 직접 조회
  • 기간: 전주 월~일 vs 전전주 월~일 성장률

✅ 판매 요약 대시보드 별도 페이지

  • 파일: dashboard_api.py → get_sales_summary_data(), sales-summary.html
  • 내용: 2025-03-01부터 어제까지 전체 OUTCOME_SALES 캠페인의 총 광고비 + 가중평균 ROAS
  • 수집: collect_sales_summary.py가 매일 실행, sales_cumulative 테이블에 누적 저장
D

수강생에게 광고계정 + 믹스패널을 받는 이유

광고계정 (Meta Ads Account)

Meta 광고에서 벌어지는 일: 노출 → 클릭 → (자사몰으로 이동)

Meta API가 주는 데이터:

  • 얼마나 노출됐는가 (impressions, reach)
  • 얼마나 클릭했는가 (clicks, CTR)
  • 광고를 얼마나 깊이 봤는가 (영상 3초/ThruPlay/지속 지표)
  • 반응이 얼마나 있었는가 (좋아요·댓글·공유·저장)
  • Meta 픽셀 기준 전환 (장바구니·결제시작·구매)
  • Meta의 상대적 품질 평가 (quality/engagement/conversion ranking)

이 데이터만으로는 자사몰 내부에서 뭐가 문제인지 모른다.
(예: 클릭은 많은데 구매가 없다 → LP 문제인지, 결제 흐름 문제인지 알 수 없음)

믹스패널 (Mixpanel)

자사몰 방문 후 벌어지는 일: LP 도착 → 스크롤 → 장바구니 → 결제 → 구매

Mixpanel이 주는 데이터 (utm_source로 Meta 트래픽만 필터링):

  • LP 도달 후 몇 초 만에 떠나는가 (이탈률 1초/10초)
  • 실제로 콘텐츠를 읽는가 (스크롤 깊이)
  • 얼마나 머무는가 (체류 시간 중앙값)
  • 자사몰 전환 퍼널 (LP → 장바구니 → 결제시작 → 구매)

광고는 좋은데 LP가 문제라면 → 스크롤이 안 되고 이탈률이 높음
LP는 괜찮은데 결제 흐름이 문제라면 → 장바구니는 많은데 결제시작/구매 전환이 낮음

코드에서의 구현

# collect_daily.py - 두 데이터를 별도로 수집
meta_rows = collect_account_meta_data(account_id, date_preset="yesterday")  # Meta API
lp_data = collect_account_lp_data(account_id, days=1)  # Mixpanel API

# 저장
save_daily_ads_to_bigquery(meta_rows)   # daily_ad_insights
save_daily_lp_to_bigquery(lp_data)      # daily_lp_metrics

퍼널 파이프라인이 완성되면 두 데이터가 결합된다:

  • CTR (Meta 데이터) → 클릭 단계의 유입량
  • 10초 체류율 (Mixpanel 데이터) → LP 도달 후 실제 콘텐츠 소비
  • 결제시작율 (Mixpanel 데이터) → 구매 의향 단계
  • 구매율 (Meta 추정 or Mixpanel 직접) → 최종 전환

dashboard_api.py에서 Meta 기반 meta_purchase_estimated 계산:

# Meta 결제시작 × 전체 구매율 = Meta 채널 구매 추정
meta_purchase_estimated = meta_checkout_users * total_purchase_rate

이게 총가치각도기의 핵심:
"광고 품질이 높아도 자사몰 LP가 나쁘면 ROAS는 낮다"를 단계별로 시각화하는 것.

개선점 (코드 확인 기반)

1. LP 섹션 비활성화 — 복원 필요

  • 현황: dynamic.html에 LP 섹션이 주석 처리됨 (// LP 품질/전환율 섹션 숨김 (데이터 정합성 이슈로 비활성화))
  • 실제 데이터: daily_lp_metrics 테이블에 정상 수집 중, API도 lp_quality 반환 중
  • 이탈률 V7 수정: CLAUDE.md에 "음수 이탈률 문제 V7에서 해결"이라고 기록되어 있음
  • 결론: 이미 수정된 V7 로직으로 수집된 데이터를 표시하도록 프론트엔드 주석만 해제하면 됨

2. 퍼널 파이프라인 데이터 연결 — 백엔드 보완 필요

  • 현황: dynamic.html에 renderFunnelPipeline() 함수와 CSS가 완비됨
  • 문제: dashboard_api.py의 날짜 범위 API 응답에 funnel 키가 없음
  • 필요 작업: daily_lp_metrics + daily_ad_insights에서 4단계 퍼널 데이터 계산 후 API 응답에 추가

3. CATALOG 벤치마크 샘플 부족

  • 현황: CATALOG 광고 5개 미만 시 IMAGE 벤치마크로 fallback
  • 한계: 카탈로그 광고가 많은 수강생은 정확한 벤치마크를 못 받을 수 있음

4. QA Helpdesk 포팅 시 고려사항

현재 아키텍처가 GCP (Cloud Run + BigQuery + GCS)에 완전히 묶여 있어, QA Helpdesk(Vercel/Supabase 기반)로 이전 시 다음 재설계 필요:

  • BigQuery → Supabase (PostgreSQL): 테이블 6개 마이그레이션
  • Cloud Run → Vercel API Routes 또는 별도 서버: Flask 앱 포팅
  • Cloud Scheduler → 외부 cron (e.g., Vercel Cron): 일일/주간 수집 트리거
  • 서비스 계정 JSON 인증 → 새 방식: GCP 서비스 계정 키가 현재 gcp-service-account.json에 있음
  • Meta 토큰 갱신 주기: 현재 장기 토큰으로 운영 중이나 만료 처리 미구현
  • 계정 인증 없음: 현재 광고계정 ID만 알면 누구나 접근 가능 (교육용이라 의도적 설계로 보임)

5. Meta ROAS vs Mixpanel ROAS 불일치

  • 피드백반 순위에서 "자사몰 매출 = Mixpanel payment_amount 합계"와 "광고매출 = Meta API purchase_value"를 별도 추적
  • 두 수치가 다를 수 있으나 (픽셀 기여 방식 vs 실결제 금액) 현재는 각각 별도 표시로 처리
  • 통합 분석 기준 정의가 없어 수강생이 두 수치 중 어느 걸 신뢰해야 하는지 가이드 없음

R1. 가중 점수 검색 알고리즘 타당성

현재 제안 분석

-- 설계서 제안 final_score = similarity + tier_boost -- tier_boost 값 Tier 1A: +0.15 (강의, 블루프린트) Tier 1B: +0.10 (과제, 피드백) Tier 2: +0.05 (Q&A, 미팅) Tier 3: +0.00 (크롤링, 유튜브)
Good

Additive 방식의 수학적 타당성은 성립

cosine similarity 범위가 [0, 1]이고 boost가 최대 0.15일 때, 정렬 순서에 미치는 영향을 분석하면:

  • 같은 Tier 내: similarity 순서 그대로 유지 (정확)
  • Tier 1A vs Tier 3: Tier 3이 이기려면 similarity 차이가 0.15 초과 필요 (합리적 임계)
  • Tier 1A vs Tier 1B: 0.05 차이 — 미세한 선호도 반영 (적절)

설계서의 예시(섹션 6)에서 Tier 1A(0.84+0.15=0.99) > Tier 3(0.91+0.00=0.91) 결과는 의도대로 작동합니다. ORDER BY priority ASC의 "무조건 Tier 1 우선" 문제를 해결한 점은 좋은 판단입니다.

F-04: 검증 범위 10개 질문은 통계적으로 불충분

권장:

  • 특히 edge case 포함: 강의에 없는 최신 주제, 여러 Tier에 걸치는 주제

F-11: Additive vs Multiplicative 대안 검토

현재 Additive 방식 외에 Multiplicative 방식도 고려할 만합니다:

-- 대안 A: Multiplicative (비율 기반) final_score = similarity * (1 + tier_weight) -- Tier 1A: sim * 1.15, Tier 3: sim * 1.00 -- 장점: 높은 similarity일수록 boost 효과 증가 -- 단점: 낮은 similarity에서 boost 효과 미미 -- 대안 B: Tunable Hybrid (향후 확장) final_score = alpha * similarity + (1 - alpha) * tier_score -- alpha = 0.8이면 similarity 80%, tier 20% 반영 -- 장점: alpha를 DB config로 관리하면 실시간 튜닝 가능

권장: 현재 Additive 방식으로 P0 진행하되, boost 값을 SQL 상수가 아닌 config 테이블에서 읽어오도록 설계하면 A/B 테스트와 튜닝이 용이합니다.

Info

F-12: 장기적으로 Hybrid Search 검토

rag-architecture.md 스킬에서 권장하는 Hybrid Search 패턴:

-- Vector + BM25 (키워드) 결합 final_score = alpha * vector_score + (1 - alpha) * bm25_score + tier_boost

메타 광고 도메인 특성상 "CBO", "ROAS", "카탈로그 광고" 같은 정확한 용어 매칭이 중요합니다. 현재 vector-only 검색은 유의어에는 강하나, 정확한 기술 용어에서는 BM25가 더 정확할 수 있습니다. P2 이후 고려사항으로 기록해 두는 것을 권장합니다.

R1 결론: 가중 점수 알고리즘은 타당합니다. boost 값의 외부 설정화와 평가 데이터셋 확대만 보강하면 P0 진행 가능합니다.

F-01: 테이블 Rename과 코드 배포의 원자성 미보장

설계서는 ALTER TABLE lecture_chunks RENAME TO knowledge_chunks를 제안하지만, DB rename과 코드 배포 사이의 갭에 대한 전략이 없습니다.

  • DB rename 실행 즉시 match_lecture_chunks RPC가 "relation does not exist" 에러 발생
  • Vercel 배포는 ~30-60초 걸리므로, 그 사이 모든 QA 검색이 실패
  • Supabase migration은 즉시 적용되나, Next.js 배포는 비동기

권장 해결책:

-- Step 1: rename 전에 VIEW 생성 (기존 코드 호환) ALTER TABLE lecture_chunks RENAME TO knowledge_chunks; CREATE VIEW lecture_chunks AS SELECT * FROM knowledge_chunks; -- Step 2: match_lecture_chunks를 search_knowledge 래퍼로 유지 CREATE OR REPLACE FUNCTION match_lecture_chunks(...) RETURNS ... AS $$ SELECT * FROM search_knowledge(...); $$; -- Step 3: 코드 배포 완료 후 (1주 뒤) VIEW/래퍼 제거

이 방식으로 zero-downtime migration이 가능합니다.

R3. 멀티모달 RAG 구현 가능성

Critical

F-03: image_embedding이 실제로는 "텍스트-about-이미지" 임베딩

설계서의 멀티모달 프로세스를 분석하면:

이미지 → Gemini Vision (이미지→텍스트 설명) → gemini-embedding-001 (텍스트→벡터) ↓ image_embedding 컬럼에 저장 (768d)

이 과정에서 저장되는 벡터는 "이미지의 텍스트 설명"을 임베딩한 것이지, 이미지 자체의 visual 임베딩이 아닙니다.

문제점:

  • image_embedding은 텍스트 임베딩과 같은 모델(gemini-embedding-001)로 생성 → 같은 벡터 공간에 이미 존재
  • 따라서 별도 image_embedding 컬럼은 중복 저장. image_description을 content에 concatenate하면 동일 효과
  • 별도 HNSW 인덱스(P2 계획)는 비용만 증가시킴
  • "동시 검색(텍스트 + 이미지)" 아키텍처가 실제로는 "같은 벡터 공간에서 두 번 검색"이 됨

권장 대안: 단순화된 이미지 처리

-- 권장: image_embedding 컬럼 제거, 대신 content에 설명 포함 -- 이미지 업로드 시: -- 1. Gemini Vision → image_description 생성 -- 2. content = original_text + "\n\n[이미지 설명] " + image_description -- 3. embedding = generateEmbedding(content) ← 하나의 벡터만 -- 4. image_url에 Storage 경로 저장 ALTER TABLE knowledge_chunks ADD COLUMN image_url TEXT, ADD COLUMN image_description TEXT; -- image_embedding 컬럼은 불필요

진정한 멀티모달이 필요한 시점

상황현재 제안권장
수강생이 스크린샷 업로드 Vision → 설명 → 별도 임베딩 Vision → 설명 → content에 합치기
강의 슬라이드 이미지 Vision → 설명 → 별도 임베딩 Vision → 설명 → content에 합치기
이미지-to-이미지 검색 image_embedding 간 비교 텍스트 설명 간 비교로 충분 (P2 범위)
진짜 멀티모달 임베딩이 필요하면: Gemini의 multimodalEmbedding API가 이미지를 직접 벡터 공간에 배치하는 기능을 지원합니다. 이 경우 텍스트-이미지 cross-modal 검색이 가능하지만, 현재 BS CAMP의 use case(~100 검색/일)에서는 과잉 설계입니다. 향후 이미지 기반 질문이 전체의 20% 이상일 때 검토를 권장합니다.
R3 결론: image_embedding 컬럼과 이미지 전용 HNSW 인덱스를 제거하고, image_description을 content에 합치는 단순 방식으로 변경하면 동일한 검색 품질을 유지하면서 스키마와 인덱스 비용을 절약합니다.

R4. HNSW 인덱스 전략

Minor

F-08: 2000행에서 HNSW는 과잉 최적화 가능성

pgvector의 순차 스캔 성능 벤치마크:

행 수Sequential ScanHNSW판단
664 (현재)~10ms~5msHNSW 불필요
2,000 (예상)~25ms~8msHNSW 선택사항
5,000~80ms~10msHNSW 권장
10,000+~200ms+~12msHNSW 필수

현재 ~100 검색/일이면 latency p95가 100ms 이내여도 충분합니다. HNSW 인덱스는 INSERT 성능을 저하시키고(~2x slower), 메모리 사용량을 증가시킵니다.

권장: P0에서 인덱스 생성 대신, P1 완료 후 2000행 달성 시점에 생성. 단, CREATE INDEX CONCURRENTLY 사용 필수.

Minor

F-09: ef_construction=64는 recall 저하 가능

설계서의 HNSW 파라미터:

CREATE INDEX ... USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);

pgvector 권장값과 비교:

파라미터설계서권장값비고
m1616적절 (10K 이하)
ef_construction64128-20064는 빌드 빠르나 recall 95% 수준. 128이면 recall 99%
ef_search (query time)미언급100쿼리 시 정확도에 직접 영향

권장: ef_construction=128로 상향. 빌드 시간이 2배 늘어나지만 2000행에서 수 초 차이. ef_search도 RPC 내에서 SET LOCAL hnsw.ef_search = 100; 추가.

R4 결론: HNSW 인덱스 자체는 올바른 방향이나, 생성 시점을 P1 이후로 연기하고, 파라미터를 상향(ef_construction=128, ef_search=100) 조정하면 더 효율적입니다.

F-07: P0 롤백 계획 부재

설계서의 P0 항목 6개 중 어느 것도 실패 시 롤백 방법을 명시하지 않습니다. 특히:

  • 테이블 rename 실패 → ALTER TABLE knowledge_chunks RENAME TO lecture_chunks;로 복원 가능하나, 새 컬럼이 이미 추가된 상태면 복잡해짐
  • search_knowledge RPC에 버그 → match_lecture_chunks 롤백이 가능하려면 래퍼가 필요 (F-01 해결책과 연계)
  • 기존 664 청크 priority UPDATE 실패 → 부분 업데이트 상태에서 검색 품질 왜곡

권장 P0 실행 계획:

-- P0 실행 순서 (트랜잭션으로 묶기) -- 0. 사전 조건 -- □ Supabase DB 백업 (Dashboard → Settings → Backups) -- □ 현재 QA 검색 품질 기준값 기록 (10개 질문 결과 캡처) BEGIN; -- 1. 테이블 rename ALTER TABLE lecture_chunks RENAME TO knowledge_chunks; -- 2. 호환 VIEW 생성 CREATE VIEW lecture_chunks AS SELECT * FROM knowledge_chunks; -- 3. 새 컬럼 추가 (모두 nullable) ALTER TABLE knowledge_chunks ADD COLUMN priority int DEFAULT 4, ADD COLUMN topic_tags text[], ADD COLUMN source_ref text, ADD COLUMN content_id uuid, ADD COLUMN chunk_total int, ADD COLUMN image_url text, ADD COLUMN image_description text; -- 4. 기존 데이터 priority 설정 UPDATE knowledge_chunks SET priority = 1 WHERE source_type = 'lecture'; UPDATE knowledge_chunks SET priority = 3 WHERE source_type = 'meeting'; -- 5. search_knowledge RPC 생성 -- (설계서 섹션 9의 SQL 그대로) -- 6. match_lecture_chunks를 래퍼로 교체 -- (기존 시그니처 유지, 내부에서 search_knowledge 호출) COMMIT; -- 7. 코드 배포 (knowledge.ts, rag.ts에서 search_knowledge 사용) -- 8. Smoke test: 10개 질문 실행, 기준값과 비교 -- 9. 1주 후: VIEW + 래퍼 제거 -- 롤백 (문제 발생 시): -- DROP VIEW IF EXISTS lecture_chunks; -- ALTER TABLE knowledge_chunks RENAME TO lecture_chunks; -- 기존 match_lecture_chunks RPC 복원

Phase별 리스크 매트릭스

Phase리스크확률영향대응
P0 DB rename과 코드 배포 갭 높음 QA 검색 전면 장애 VIEW + 래퍼 패턴 (F-01)
supabase.ts 타입 불일치 높음 빌드 실패 타입 재생성 (F-02)
priority UPDATE 부분 실패 낮음 검색 순위 왜곡 트랜잭션으로 묶기
P1 재임베딩 중 벡터 불일치 중간 일시적 검색 품질 저하 새 컬럼에 저장 후 swap
Gemini Rate Limit 중간 파이프라인 중단 Exponential backoff (설계서 있음)
P2 외부 API 의존성 (Notion, yt-dlp) 중간 소스 수집 실패 각 소스 독립 실행 + retry
피드백반 개인정보 노출 높음 프라이버시 위반 익명화 선처리 (설계서에서 인지함)
P1 재임베딩 관련 추가 의견: 설계서에서 "기존 벡터로 정상 운영"이라 했으나, 모델이 동일(gemini-embedding-001)하고 outputDimensionality도 768로 동일하면 재임베딩이 불필요할 수 있습니다. 현재 gemini.ts에서 이미 gemini-embedding-001을 사용 중이므로, 기존 664개 벡터는 그대로 호환됩니다. 재임베딩은 모델이나 차원이 변경될 때만 필요합니다.
R6 결론: 마이그레이션 순서는 합리적(P0 기반→P1 파이프라인→P2 확장)이나, 각 Phase의 rollback SQL, smoke test, 성공 기준을 추가해야 합니다. 특히 P0는 반드시 트랜잭션으로 묶어 원자성을 보장해야 합니다.

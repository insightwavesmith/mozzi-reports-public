❓ 왜 "무조건 Tier 1 먼저"가 아닌가?
Smith님의 원안 ORDER BY priority ASC는 Tier 1에 유사도 0.71인 약한 매칭이 Tier 3의 0.95 강한 매칭보다 무조건 먼저 나옵니다.

🎯 가중 점수 방식은 Tier 1에 가산점을 주되, 정말 관련없는 내용이 올라오는 것은 방지합니다.
예: Tier 1 (0.71 + 0.15 = 0.86) vs Tier 3 (0.95 + 0.00 = 0.95) → Tier 3이 더 관련성 높으면 Tier 3가 우선

4. 데이터가 흘러가는 과정 (비개발자용)

🏭 "자동차 공장" 비유
원자재(소스) → 부품 가공(청킹) → 조립라인(임베딩) → 완성품(검색 가능한 지식) → 고객(5개 Consumer)에게 배송
┌─────────────────── 데이터 파이프라인 ───────────────────┐ │ │ │ [8종 소스] ──→ [contents] ──→ [청킹] ──→ [임베딩] │ │ │ │ │ │ │ │ │ │ │ ▼ │ │ 수집 방식 원본 저장 의미 단위 좌표 변환 │ │ 자동/수동 메타데이터 분할 768개 숫자 │ │ │ │ │ │ │ │ ▼ ▼ ▼ ▼ │ │ │ │ 강의: 관리자 텍스트+이미지 주제별 "의미의 지도" │ │ 크롤링: 크론 +우선순위 800자 비슷한 의미= │ │ 유튜브: yt-dlp +태그 +100겹침 가까운 좌표 │ │ Q&A: 실시간 +URL 1쌍=1청크 │ │ Notion: API +embedded_at 과제=1청크 │ │ │ │ ──→ [knowledge_chunks] ──→ [search_knowledge RPC] │ │ │ │ │ │ 벡터 저장 가중 점수 │ │ +메타데이터 검색 알고리즘 │ │ +이미지 URL │ │ │ │ ▼ │ │ └──────────────→ [5개 Consumer] │ │ 답변 │ └──────────────────────────────────────────────────────┘

📋 단계별 세부 설명

1
소스 수집 단계
각 소스가 contents 테이블에 저장됩니다:
  • 강의: 관리자가 수동 업로드, 기수별로 구분 관리
  • 크롤링: 크론 작업으로 매일 자동 수집
  • 유튜브: yt-dlp로 자막 자동 추출
  • Q&A: 답변 완료 시 실시간 자동 저장
  • Notion: API를 통한 주기적 동기화
2
청킹 (의미 단위로 분할)
소스별로 다른 전략으로 검색하기 좋은 크기로 나눕니다:
  • 강의: 주제/슬라이드 단위 (주제 맥락 보존)
  • 과제: 1과제=1청크 (쪼개면 의미 손실)
  • Q&A: 질문+답변 1쌍=1청크 (완결된 대화)
  • 크롤링: 800자 + 100자 겹침
🧩 "겹침"이 뭔가요?
책을 찢어서 나눈다면 문장이 중간에 끊어질 수 있습니다. "겹침"은 앞 페이지의 마지막 100자를 다음 페이지 시작에 포함시켜서, 경계에서 맥락이 끊기는 것을 방지하는 기법입니다.
3
임베딩 (텍스트→숫자 좌표 변환)
Gemini gemini-embedding-001을 사용해 텍스트를 768개 숫자로 변환합니다.
🗺️ "의미의 지도" 비유
"페이스북 광고"와 "Meta 캠페인"은 비슷한 의미이므로 지도상에서 가까운 위치에 놓입니다. "광고 예산"과 "요리 레시피"는 전혀 다른 의미이므로 멀리 떨어진 위치에 놓입니다.
월 비용: ~$2 (검색량에 따라 변동)
4
벡터 저장 → 검색 가능
knowledge_chunks 테이블에 벡터와 메타데이터가 함께 저장되어 "의미 기반 검색"이 가능해집니다. 단순 키워드 매칭이 아닌, 진짜 의미가 비슷한 내용을 찾을 수 있습니다.
5
Consumer가 활용
5개 서비스(QA 헬프데스크, 블로그, 뉴스레터, 웨비나, 챗봇)가 모두 같은 search_knowledge RPC를 호출해서 지식을 활용합니다.

🔍 [F-03 반영] 이미지 검색 방식 — 별도 벡터 없이 텍스트 통합
Gemini는 이미지를 직접 벡터로 변환하지 못합니다. 대신:

장점: 별도 image_embedding 컬럼/인덱스 불필요. 기존 검색 파이프라인 그대로 활용.
단점: 이미지의 시각적 유사도(색상, 레이아웃)는 못 잡음. 의미적 유사도만 가능.

-- 핵심 공식 final_score = (1 - cosine_distance) + tier_boost -- tier_boost 값: -- Tier 1A (강의, 블루프린트): +0.15 -- Tier 1B (과제, 피드백): +0.10 -- Tier 2 (Q&A, 미팅): +0.05 -- Tier 3 (크롤링, 유튜브): +0.00 -- 정렬: ORDER BY final_score DESC

핵심 변경 4: search_knowledge RPC

CREATE OR REPLACE FUNCTION search_knowledge( query_embedding vector(768), match_threshold float DEFAULT 0.4, match_count int DEFAULT 10, filter_source_types text[] DEFAULT NULL ) RETURNS TABLE ( id uuid, content text, similarity float, tier_boost float, final_score float, source_type text, priority int, lecture_name text, topic_tags text[], source_ref text, image_url text ) LANGUAGE SQL AS $$ SELECT kc.id, kc.content, 1 - (kc.embedding <=> query_embedding) AS similarity, CASE kc.priority WHEN 1 THEN 0.15 WHEN 2 THEN 0.10 WHEN 3 THEN 0.05 ELSE 0.00 END AS tier_boost, (1 - (kc.embedding <=> query_embedding)) + CASE kc.priority WHEN 1 THEN 0.15 WHEN 2 THEN 0.10 WHEN 3 THEN 0.05 ELSE 0.00 END AS final_score, kc.source_type, kc.priority, kc.lecture_name, kc.topic_tags, kc.source_ref, kc.image_url FROM knowledge_chunks kc WHERE 1 - (kc.embedding <=> query_embedding) > match_threshold AND (filter_source_types IS NULL OR kc.source_type = ANY(filter_source_types)) ORDER BY final_score DESC LIMIT match_count; $$;

핵심 변경 5: HNSW 인덱스 (벡터 검색 성능)

왜 필요한가: 2000+ 청크가 되면 벡터 검색이 느려질 수 있습니다. HNSW 인덱스는 "빠른 근사 검색"을 가능하게 합니다. 정확도 99%를 유지하면서 속도를 10배 이상 높입니다.
-- 벡터 검색 성능 인덱스 (P0에서 생성) CREATE INDEX idx_knowledge_chunks_embedding ON knowledge_chunks USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64); -- [F-03] image_embedding 인덱스 불필요 -- 이미지는 Gemini Vision → 설명 텍스트 → content에 합침 → 기존 embedding 벡터로 검색

재임베딩 배치 전략: ├── 배치 크기: 10개/요청 ├── 요청 간 대기: 1초 ├── 실패 시 백오프: 2초 → 4초 → 8초 (3회 재시도) ├── 예상 소요: 664개 ÷ 10 = 67배치 × 1초 ≈ 2분 ├── 재임베딩 중 서비스: 기존 벡터로 정상 운영 └── 완료 후: 새 벡터로 교체 + 품질 테스트

10. 주요 결정 사항 (ADR)

ADR-1: 단일 벡터 테이블 유지

결정: knowledge_chunks 하나에 모든 소스 저장 (테이블 분리 안 함)

이유: 664개 + 예상 ~2000개 = 스케일 문제 없음. 단일 테이블이 검색 단순.

리스크: 5000+ 넘으면 소스별 분리 검토. HNSW 인덱스 추가 필요할 수 있음.

ADR-2: Gemini 임베딩 유지

결정: gemini-embedding-001 (768d) 유지, 로컬(bge-m3) 전환 안 함

이유: 이미 사용 중, 한국어 품질 충분, $2/월, 이미지 임베딩도 지원.

리스크: Gemini API 장애 시 임베딩/검색 불가 → 로컬 fallback 향후 검토.

ADR-3: 가중 점수 검색 (ORDER BY priority ASC 대신)

결정: final_score = similarity + tier_boost 방식

이유: 순수 우선순위 정렬은 약한 Tier1이 강한 Tier3를 밀어냄. 가중 점수는 Tier1 가산점 + 관련성 보장.

리스크: boost 값 튜닝 필요. 기존 QA 질문 10개로 비교 테스트 후 확정.

ADR-4: Tier 1A/1B 분리

결정: 메타 블루프린트 + 자사몰사관학교 = Tier 1A (최우선), 과제+피드백 = Tier 1B

이유: "근거가 명확하고 자료가 가장 많은" 두 소스. 공식 교육 자료와 자체 교육 자료가 수강생에게 가장 신뢰도 높음.

리스크: 블루프린트 자료 수집 방법 확정 필요. 크롤링 가능한지 or 수동 업로드만인지.

ADR-5: 멀티모달 RAG — 텍스트 통합 방식 [F-03 반영]

결정: Gemini Vision → 설명 텍스트 → content에 합쳐서 텍스트 임베딩. image_embedding 별도 컬럼 없음.

이유: Gemini embedding API는 이미지 직접 벡터화를 지원하지 않음. 설명 텍스트를 content에 합치면 기존 파이프라인 변경 없이 이미지 검색 가능.

리스크: 시각적 유사도(색상/레이아웃)는 못 잡음, 의미적 유사도만 가능. Vision 처리 시간 ~2초.

ADR-6: Hybrid Search 도입 (벡터 + 키워드 병행)

결정: P1에서 벡터 검색 + PostgreSQL Full-Text Search(tsvector) 병행. final_score = α × vector_score + (1-α) × bm25_score + tier_boost

이유: "메타 광고", "ROAS", "전환 API" 같은 정확한 용어 검색은 벡터만으로 약함. 키워드 매칭을 병행하면 정확한 용어가 포함된 청크가 확실히 올라옴.

구현: knowledge_chunks에 search_vector tsvector 컬럼 추가 + GIN 인덱스. search_knowledge RPC에서 벡터 유사도와 ts_rank를 합산.

리스크: α 값 튜닝 필요 (권장 시작값 0.6). 한국어 형태소 분석기 설정 필요.

ADR-7: Reranking 검토 (P2)

결정: P2에서 Cohere Rerank 또는 Cross-encoder 도입 검토. 초기 검색 top-20 → 재정렬 → top-5 반환.

이유: 소스가 8종으로 늘어나면 초기 검색 정확도만으로 부족할 수 있음. 재정렬은 "진짜 관련 있는 것"을 한 번 더 걸러줌.

리스크: 추가 API 호출 비용 + 200ms 지연. 현재 규모에서는 과도할 수 있으므로 P2 이후 데이터 평가 결과 보고 결정.

ADR-8: 검색 품질 평가 체계 (Gold Standard Set)

결정: P1에서 최소 50개 Gold Standard 질문-답변-소스 세트 구축. Recall@5 > 0.8, MRR > 0.5 목표.

이유: "QA 질문 10개 테스트"는 통계적으로 불충분. 변경 전후 비교, A/B 테스트, 튜닝 효과 측정에 최소 50개 필요.

구축 방식:

  • 기존 QA 이력에서 대표 질문 30개 선별
  • Smith님이 "이 질문에는 이 강의 내용이 나와야 한다" 정답 매핑
  • 엣지 케이스 20개 추가 (다중 소스 관련, 이미지 포함, 애매한 질문)
  • 분기별 업데이트
리스크: 정답 매핑에 Smith님 시간 필요. 자동 생성(LLM)은 품질 보장 어려움.

ADR-9: 검색 품질 모니터링 대시보드

결정: P1에서 knowledge_usage 테이블에 모니터링 지표 추가, P2에서 대시보드 시각화.

추적 지표:

지표설명알림 기준
avg_similarity검색 결과 평균 유사도< 0.5 → 임베딩 품질 점검
empty_result_rate검색 결과 0건 비율> 5% → threshold 하향 또는 데이터 보강
retrieval_latency_p95검색 응답시간 95퍼센타일> 3초 → 인덱스 점검
source_distribution검색 결과의 소스별 비율Tier 3만 나오면 데이터 부족 신호
user_feedback답변 만족도 (👍/👎)👎 > 20% → 검색 튜닝 필요
리스크: 로깅 오버헤드 미미 (JSON 1행/쿼리). 대시보드는 기존 mozzi-reports에 통합 가능.

ADR-10: 디버깅 플레이북

결정: 검색 품질 이슈 발생 시 따르는 표준 진단 절차 수립
증상진단 순서
엉뚱한 문서 검색됨 ① 질문 임베딩 품질 확인 (너무 짧거나 모호?) → ② 해당 청크 내용 점검 (잘못 청킹?) → ③ Hybrid Search 키워드 매칭 추가 → ④ tier_boost 값 조정
관련 문서가 있는데 안 나옴 ① 해당 문서가 임베딩됐는지 확인 (embedded_at) → ② threshold 낮춰서 테스트 (0.4→0.3) → ③ top_k 늘려서 확인 → ④ 청크에 핵심 키워드가 포함됐는지 점검
전반적으로 유사도 낮음 ① 임베딩 모델 변경됐는지 확인 → ② 도메인 용어가 임베딩에 반영되는지 테스트 → ③ 재임베딩 검토
Tier 3만 나옴 ① Tier 1A/1B 데이터 부족 신호 → ② 해당 주제의 강의 청크 존재 여부 확인 → ③ 강의 자료 보강 필요

ADR-11: Query Expansion (질문 확장)

결정: P2에서 도입. 애매한 질문을 구체적 검색어로 확장 후 검색.

비유: 도서관에서 "광고 안 돼요"라고 하면 사서가 "어떤 광고요? 메타? 구글? 성과가 안 나온다는 건가요?"라고 되물어서 정확한 책을 찾아주는 것.

구현:

  • "광고가 안 돼요" → LLM이 확장 → "메타 광고 성과 저하 원인 진단 ROAS 하락"
  • 원본 질문 + 확장 질문 모두로 검색 → 결과 합산
  • 비용: 확장용 LLM 호출 1회 추가 (Sonnet으로 처리하면 ~$0.01/건)
리스크: 확장이 잘못되면 오히려 노이즈 증가. 확장 전후 검색 품질 비교 필수.

ADR-12: A/B 테스트 프레임워크

결정: P2에서 도입. 검색 설정 변경 시 Gold Standard Set으로 전후 비교.

테스트 대상:

변수테스트 범위측정 지표
청크 크기400 vs 800 vs 1200자Recall@5, MRR
겹침 비율0% vs 10% vs 20%Recall@5
Hybrid α 값0.5 vs 0.6 vs 0.7MRR, Precision@5
tier_boost 값현재 vs ×2 vs ×0.5소스 분포, 만족도
threshold0.3 vs 0.4 vs 0.5empty_result_rate, noise

최소 샘플: Gold Standard 50개 기준. 통계적 유의미성을 위해 각 설정별 전체 50개 실행 후 비교.

리스크: 테스트 시 임베딩 비용 발생 (청크 크기 변경 시 재임베딩 필요). 한 번에 하나씩 변경 원칙.
